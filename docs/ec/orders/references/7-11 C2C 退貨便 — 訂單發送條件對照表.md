商家在訂單列表勾選訂單後，系統會依下列條件判斷該批次動作是否可用。任一條件不符即無法發送退貨代碼。

## 基本條件 { #reference-seven-eleven-c2c-return-eligibility-basic }

| 條件 | 規則 | 不符時的處理 |
| :-- | :-- | :-- |
| 加值功能開通 | 商店已申請開通「7-ELEVEN C2C 退貨便」 | 批次動作選單中不會出現此選項 |
| 訂單金額 | 單筆訂單總金額需 **小於 NT$20,000** | 該訂單會被排除，無法發送 |
| 配送方式 | 訂單僅使用單一配送方式 | 多重配送方式(拆單)的訂單無法批次發送 |
| 是否已申請過退貨 | 訂單未處於「退貨審查」或「已退貨」狀態 | 一筆訂單僅能申請一次退貨 |


## 配送方式 × 配送進度 矩陣 { #reference-seven-eleven-c2c-return-eligibility-matrix }

訂單需符合下列任一組合，才能發送退貨代碼：

| 配送方式 | 可發送的配送進度 | 說明 |
| :-- | :-- | :-- |
| **CYBERBIZ 物流** | 已出貨 / 已到貨 / 已收貨 | 包含黑貓、新竹、宅配通、超商取貨等由 CYBERBIZ 串接的物流 |
| **其他物流** | 僅限「已收貨」 | 商家自行寄送、未經 CYBERBIZ 串接的物流，需確認消費者已收到才能發送退貨代碼 |

!!! info "為什麼有此差異"
    其他物流由於系統無法追蹤實際配送進度，僅在商家手動標記「已收貨」後才確認消費者已實際收到商品，避免在配送途中誤發退貨代碼造成包裹遺失。

## 退貨狀態條件 { #reference-seven-eleven-c2c-return-eligibility-return-status }

訂單的退貨狀態須為下列三者之一：

| 退貨狀態 | 對應情境 |
| :-- | :-- |
| **不需退貨** | 一般訂單預設狀態，消費者尚未提出退貨申請 |
| **申請退貨** | 消費者已在會員中心提出退貨申請，等待商家回應 |
| **退貨中** | 已發送過退貨代碼(此情境用於補發或重發代碼) |

!!! note "註釋"
    若訂單已進入「退貨審查」或「已退貨」階段，系統會視為退貨流程已啟動或完成，不允許再次發送代碼。如需重新發送(例如代碼已過期)，請先聯繫 CYBERBIZ 客服協助處理訂單狀態。

## 常見問題

??? quote "哪些情況下訂單無法使用 7-11 C2C 退貨便功能？"
    [](){ #faq-seven-eleven-c2c-return-eligibility-basic-unmet }

    下列任一條件不符即無法發送退貨代碼：

    - [x] **訂單金額超過 NT$20,000**：超過 7-11 物流系統的硬性限制，需改用其他退貨方式
    - [x] **訂單使用多重配送方式**：僅支援單一配送方式的訂單，拆單訂單無法批次發送
    - [x] **訂單已處於「退貨審查」或「已退貨」**：一筆訂單僅能申請一次退貨代碼
    - [x] **未開通加值功能**：商店需先向 CYBERBIZ 客服申請開通，否則選單中不會出現此選項

??? quote "為什麼 CYBERBIZ 物流與其他物流的配送進度條件不同？"
    [](){ #faq-seven-eleven-c2c-return-eligibility-logistics-diff }

    這是因為系統追蹤能力的差異：

    - **CYBERBIZ 物流**（黑貓、新竹、宅配通、超商取貨等）：系統可即時追蹤配送進度，因此支援 **已出貨 / 已到貨 / 已收貨** 三種狀態
    - **其他物流**（商家自行寄送、未經串接）：系統無法追蹤實際配送進度，僅在商家手動標記「已收貨」後才確認商品已送達，避免配送途中誤發退貨代碼造成包裹遺失

??? quote "已發送過退貨代碼的訂單還能再補發嗎？"
    [](){ #faq-seven-eleven-c2c-return-eligibility-resend }

    可以，但需符合以下條件：

    - 訂單的退貨狀態為「退貨中」（表示已發過代碼）
    - 若代碼已過期（超過 7 天效期），可聯繫 [CYBERBIZ 客服](#){ data-preview } 協助處理訂單狀態後重新發送
    - 已進入「退貨審查」或「已退貨」階段的訂單無法再次發送
